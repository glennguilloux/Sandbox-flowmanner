BUGS FOUND (1 critical, 2 minor):

  [BUG-1 — CRITICAL: Endpoint path mismatch]
    The sub-agent's report says:
      POST /depth/decide
      GET /missions/{id}/depth-events
    The docstring in depth.py:1-6 says the same.
    ACTUAL FastAPI route table (verified by importing app.main_fastapi):
      POST /api/depth/decide
      GET /api/depth/missions/{mission_id}/depth-events
    Root cause: depth.py:24 declares `APIRouter(prefix="/depth", tags=["depth"])`
                and line 97 registers `@router.get("/missions/{mission_id}/depth-events", ...)`
                The /depth prefix concatenates → /api/depth/missions/{id}/depth-events
    Impact: Any client trying to hit /api/missions/{id}/depth-events gets 404. The endpoint
            is reachable but at a non-standard, surprising path under /api/depth/.
    Severity: CRITICAL because the report itself documents the wrong path — i.e. nobody
              could have actually used the endpoint to verify it works.
    Fix: Split depth_router into two routers, OR change the GET endpoint to a separate
         events router without the /depth prefix. E.g.:
           # depth.py
           router = APIRouter(prefix="/depth", tags=["depth"])  # POST /decide only
           events_router = APIRouter(tags=["depth-events"])  # GET /missions/{id}/depth-events
           @events_router.get("/missions/{mission_id}/depth-events", ...)
         And register both in __init__.py. About 8 lines of changes.

  [BUG-2 — MINOR: reason string joins ALL signals, not just the winner]
    In depth_policy.py:152-153:
       reason = "; ".join(reasons)
    where `reasons` is appended for EVERY signal (lines 127, 132, 137, 142). For a
    low-risk, low-uncertainty, high-budget step that picks SHALLOW, the reason is:
      "risk=low; uncertainty=0.20 < 0.3; prior_failures=0; budget=$5.00 (adequate)"
    But for a HIGH-RISK step that picks DEEP, the reason ALSO includes "uncertainty=0.20
    < 0.3" and "prior_failures=0" and "budget=$5.00 (adequate)" — the audit log
    will show the user that the policy voted DEEP because of risk, but the reason
    string implies the other 3 signals also contributed. This is misleading for replay.
    Fix: Build the reason from the winning candidate's reason + (optional) "runner-up"
         notes, or list the winning signal first and tag the others as "non-binding".

  [BUG-3 — MINOR: _emit_depth_event opens a new DB session instead of using the caller's]
    In mission_executor.py:810-851, _emit_depth_event takes `db` as a parameter but
    never uses it — instead it opens `async with AsyncSessionLocal() as event_db:`
    on line 840. The depth event is committed in a separate transaction from the
    parent mission execution. If the parent rolls back, the depth event STAYS in
    the DB. This is documented as "non-critical, log and continue" but it can cause
    audit/replay divergence.
    Fix: Either use the passed `db` session (and accept that audit fails if the
         mission transaction fails), or use a savepoint inside the parent session.
         Lowest-risk fix: add a comment documenting the divergence explicitly.
