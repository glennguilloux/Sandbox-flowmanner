## What the user actually experiences

Two distinct visible states map to two distinct failure modes:

| Visible UI text | Condition | Code path |
|----------------|-----------|-----------|
| `Loading preview…` | API in flight | line 80-87, `loading=true` |
| `Preview unavailable` (RED) | **API threw** | line 90-99, `error` set in catch on line 65 |
| `Sandbox starting…` | status in {creating, starting, provisioning} | line 104-117 |
| `Sandbox stopped` | status in {stopped, exited} | line 120-128 |
| `Open Preview` (link) | `info.preview_url && info.status === "running"` | line 132-148 |
| `Sandbox running (no dev server)` | running but `preview_url` is null | line 151-160 |

**The "live preview is not available" complaint the user reported maps to the RED "Preview unavailable" branch.** That branch has exactly one trigger: a thrown error in `getSandboxPreview` → an exception during the `GET /api/sandbox/{id}/preview` request.

## Most likely root causes (ranked)

1. **LLM skipped `sandboxd_serve`** (the workflow's only tool that returns a working URL). The sandbox exists (`sandboxd_preview` returned an id) but the dev server was never started, so `info.preview_url` is null. **This would NOT show "Preview unavailable"** — it would show "Sandbox running (no dev server)" (yellow). So this is NOT the cause of the user-reported message.

2. **Sandbox expired or was killed** between the chat-message render and the polling request. The 35-minute sandbox lifetime (per `chat_service.py:287`) is short, and `sandboxd get(sandbox_id)` returns 404. The frontend's poll gets a 404 → throws → "Preview unavailable". **Likely cause.** Especially if the user reports it after a long chat or after returning to the tab.

3. **Auth failure on the preview path.** The `get_current_user` dependency in `sandbox_preview.py:95` requires a valid session. If the user's session expired between when the chat was sent and when they tried to open the preview, the API returns 401, the frontend throws, "Preview unavailable" shows. **Plausible.** This is a known class — the auth-regression product bug is still open per task-10 drift report.

4. **sandboxd service itself down** (e.g. :9090 not reachable, or the IONOS firewall / sandboxd token expired). Then `client.get(sandbox_id)` raises a connection error → 404 (the catch-all in the route) → "Preview unavailable". **Possible** but less likely; would affect all sandboxes, not just one.

5. **sandboxd returned a malformed response** (e.g. preview field missing). The route returns `preview_url=null` → "Sandbox running (no dev server)" (yellow). **Not the cause** of the red "Preview unavailable".

## Why this is bad UX (independent of root cause)

The error message **"Preview unavailable"** carries zero actionable information:
- No status code
- No request_id
- No hint about *which* of the 4 possible failure modes occurred
- No "Retry" / "Reopen sandbox" action
- No log correlation (the `exc` from `sandboxd get(...)` is swallowed into a 404 detail that never reaches the user)

A user seeing this cannot tell whether to retry, refresh, restart the sandbox, or report a bug.

## Recommended fixes (prioritized)

### Must-do (P0.2 acceptance)

**A) Make the error actionable in the UI** — change SandboxPreviewButton.tsx:90-99 to show the actual error message + a retry button. Minimum:
```tsx
if (error) {
  return (
    <div className="... bg-red-950/30 ...">
      <AlertCircle className="h-4 w-4" />
      <span>Preview unavailable — {error}</span>   {/* show the real reason */}
      <button onClick={() => { setError(null); setLoading(true); /* refetch */ }}>Retry</button>
    </div>
  );
}
```

**B) Differentiate failure modes in the backend** — instead of one catch-all 404, return:
- 404 only when sandboxd confirms the sandbox is gone
- 502 when sandboxd is unreachable (network/timeout)
- 504 when sandboxd times out
- Log the `exc` with the `request_id` and emit a structured error so Sentry/Jaeger can correlate

This requires editing `sandbox_preview.py:111-116` and possibly adding a sandboxd-client error class.

### Nice-to-have (separate ticket)

- A "Restart sandbox" action in the UI (calls `sandboxd_serve` again with the same `sandbox_id`)
- Surface a 35-min countdown so users know when their preview will expire
- Per-sandbox error log in the workspace activity feed
