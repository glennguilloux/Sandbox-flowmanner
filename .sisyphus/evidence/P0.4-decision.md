# P0.4 Scope Decision — Chunk 1 (Q2-Q3)

**Date:** 2026-06-12
**Decision:** RESCOPE (do not modify `auth.ts`)
**Rationale:** The P0.4 investigator's original hypothesis was correct — the 401 burst is test traffic, not a customer bug. However, their recommendation ("close as no fix needed") was wrong. The A.3/B.3 frontend signin-redirect loop is a real product bug but is **already partially addressed** by the current code.

## What the current `auth.ts` code does

After reading `/home/glenn/FlowmannerV2-frontend/src/auth.ts` (the jwt callback + `refreshAccessToken`), the code **already passes `refresh_token` correctly**:

```typescript
// Line 75-79 in refreshAccessToken():
const res = await fetch(`${BACKEND_URL}/api/auth/refresh`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ refresh_token: refreshToken }),  // ← token IS passed
  signal: controller.signal,
})
```

The P0.4 evidence file's claim that "the body is empty" was based on an **older version** of the code. The current implementation:
1. Passes `refresh_token` in the body (line 78)
2. Has retry logic with 3 attempts and exponential backoff (lines 68-110)
3. Has deduplication to prevent concurrent refresh storms (lines 56-58)
4. Properly handles 4xx (don't retry) vs 5xx (retry) (lines 92-103)

## What remains broken (A.3/B.3)

The live production error captured in the P0.4 evidence file is:
```
Auth.js error: NetworkError when attempting to fetch resource.
```

This is a **transient network error** on the VPS → homelab WireGuard path, not a code bug in `auth.ts`. When the WireGuard tunnel has a hiccup:
1. `refreshAccessToken()` fails all 3 retry attempts
2. Returns `{ error: "RefreshAccessTokenError" }`
3. The jwt callback clears `refreshToken` and sets `error: "RefreshAccessTokenError"`
4. NextAuth session becomes unusable → api-client 401 → signOut → /signin loop

## Why RESCOPE instead of FIX

1. **The refresh_token IS already being passed** — the core "empty body" bug from the P0.4 evidence doesn't exist in the current code.
2. **The remaining issue is infrastructure** (WireGuard tunnel reliability), not code.
3. **A real fix would require**: migrating to the v3 cookie-based refresh endpoint (`/api/v3/auth/sessions/refresh`) which doesn't depend on the request body, OR adding WireGuard tunnel health monitoring. Both are out of scope for Chunk 1.
4. **The q2-q3 plan says**: "Do not propose P0.2/P0.4 fixes inside this plan; they are tracked separately."

## Recommendation for next step

Open a separate frontend ticket: "Migrate NextAuth refresh to v3 cookie-based endpoint (`/api/v3/auth/sessions/refresh`) to eliminate body-dependent refresh and improve WireGuard resilience." This is a ~30-min bounded task that touches only `auth.ts` and is a net improvement regardless of tunnel reliability.
