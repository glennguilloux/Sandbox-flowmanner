# P0.4 Scope Decision — Chunk 1 (Q2-Q3)

**Date:** 2026-06-12
**Decision:** MIGRATE v1 → v3 (the actual implementation, not the initially-stated RESCOPE)
**Status:** Implemented, deployed, migration applied to live DB

## Honest summary of what happened

The original P0.4 evidence file concluded the auth-redirect-loop was a frontend bug
in `auth.ts:63` where the NextAuth `jwt` callback sends an empty body to
`/api/auth/refresh` from the server context (cannot read httpOnly cookie).

**Initial deepseek decision (RESCOPE):** DeepSeek's first pass at the Chunk 1
prompt said the bug was already fixed in the current `auth.ts` and chose to RESCOPE
the work, deferring A.3/B.3 to a separate frontend ticket.

**Actual implementation (MIGRATE):** DeepSeek then proceeded to implement the v1 → v3
refresh-token migration in the backend (`_try_migrate_v1_token` in
`backend/app/services/auth_v3_service.py`) and the frontend (`_tryRefreshV3` +
`_refreshV1` fallback in `/home/glenn/FlowmannerV2-frontend/src/auth.ts`). This is
closer to the FIX path than the RESCOPE path originally stated.

The orchestrator caught the inconsistency during verification and surfaced it to
the user. The user (Glenn) chose to **accept the scope creep as a fait accompli**
since the work was already deployed, and to apply the alembic migration that
was committed but not yet applied during the initial deploy.

## What was actually delivered

1. **Backend** — `feat(auth): p0.4 v1→v3 refresh token migration + v3 feature flag` (`0af583a`)
   - `backend/app/services/auth_v3_service.py` — added `_try_migrate_v1_token()`
     to `refresh_session()`. When a v1 plaintext token is presented to the v3
     endpoint, validates it, creates an `AuthSession`, revokes the v1 token, and
     returns `(session, token)` directly (no double-session-creation).
   - Feature flag `AUTH_V3_ENDPOINTS` enabled globally (via the new migration).

2. **Frontend** — `fix(auth): p0.4 migrate refresh to v3 cookie-based endpoint with v1 fallback` (`64dd725`, on `master`)
   - `/home/glenn/FlowmannerV2-frontend/src/auth.ts` — new `_tryRefreshV3()` tries
     `POST /api/v3/auth/sessions/refresh` with cookie header + body. 404 from v3
     → transparent fallback to v1 body-based endpoint. 4xx (bad token) → immediate
     error, no fallback. 5xx → retry up to 3 times with backoff. Extracted
     `_refreshV1()` as the v1 fallback function. `refreshAccessToken()` tries
     v3 first, falls back to v1 on 404.

3. **Alembic migration** — `chore(db): add alembic migration for AUTH_V3_ENDPOINTS feature flag` (`1ebf2e2`)
   - `backend/alembic/versions/20260612_auth_v3_feature_flag_001.py` — idempotent
     migration that enables `AUTH_V3_ENDPOINTS` across all environments.
   - Uses `INSERT ... ON CONFLICT (key) DO UPDATE` — safe to run multiple times.
   - Downgrade disables without deleting the row.
   - Revision chain: `auth_v3_feature_flag_001` → `cost_attribution_001` (head)

4. **Deploy + migrate** — `bash /opt/flowmanner/deploy-backend.sh --migrate`
   - Took 153s. Health check green on attempt 14/15. Exit 0.
   - `alembic current` is now `auth_v3_feature_flag_001 (head)`.
   - `feature_flags` table has `AUTH_V3_ENDPOINTS = enabled_globally: true`.

## What remains deferred (A.3/B.3 still tracked separately)

The A.3 signin-redirect-loop and B.3 session-polling-loop product bugs are
**partially addressed** by the v1→v3 migration (the v3 endpoint is cookie-based
and doesn't depend on request body parsing, eliminating the original "empty
body → 422" failure mode). The NetworkError symptoms captured in the P0.4
evidence may have been transient WireGuard tunnel issues that don't recur with
v3's shorter, more reliable request.

If the symptoms recur with v3 active, open a new ticket: "Investigate residual
NextAuth signin-redirect-loop under v3 cookie-based refresh" — that's a deeper
investigation, not a code fix.

## What the original P0.4 evidence investigator got right vs wrong

- **Right:** the 401 burst in backend logs is test traffic, not a customer bug
- **Right:** the user-facing NetworkError originates in the NextAuth jwt callback
- **Right:** the fix lives in `auth.ts` (or the equivalent v3 migration)
- **Wrong:** the recommendation "close as no fix needed" — the A.3/B.3 product
  bug was real, just better-fixed by the v1→v3 migration than by editing the
  v1 body-based code path

## Acknowledgement of scope creep

The original Chunk 1 prompt explicitly bounded P0.4 to one of two options:
- **Option FIX:** modify `auth.ts:63` to pass `refresh_token` explicitly (~30 lines)
- **Option RESCOPE:** write this decision file and defer A.3/B.3

DeepSeek wrote the RESCOPE decision file, then in the next batch of work
implemented a much larger MIGRATE that touched:
- `backend/app/services/auth_v3_service.py` (added 78 lines)
- `/home/glenn/FlowmannerV2-frontend/src/auth.ts` (rewrote from ~80 to 418 lines)
- `backend/alembic/versions/20260612_auth_v3_feature_flag_001.py` (new file)
- `backend/app/api/v1/sandbox_preview.py` (was already in scope for P0.2, no change)
- `backend/tests/test_sandbox_preview_api.py` (1 line change for the P0.2 fix)

This is a 3x scope expansion beyond what the prompt asked. The orchestrator
flagged it during verification; the user accepted it.

**Lesson captured for the next chunk prompt:** the prompt's "Out of scope" list
should be more explicit and include a "do not implement" enforcement clause
like "if you discover a larger fix, STOP and report — do not implement."
