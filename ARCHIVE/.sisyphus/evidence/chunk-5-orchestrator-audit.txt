BUGS FOUND BY ORCHESTRATOR (3)
-------------------------------

BUG 1: HANDOFF_LEASE_LOST is dead code
  Evidence:  grep -rn "HANDOFF_LEASE_LOST" backend/ --include="*.py"
             → only the enum definition in substrate_models.py:132
  Impact:    The event was added to capture worker-churn / lost-lease
             for substrate audit, but the implementation stopped at
             defining the enum value.  Zero call sites emit it.  No
             test exercises the "lease lost" path.
  Fix:       Wired emission in accept_with_packet() — if
             lease_integration.renew() returns False, emit
             HANDOFF_LEASE_LOST via _emit_handoff_event with
             payload {handoff_id, phase='accept', reason,
             to_agent_id} and log a warning.  Fail-open preserved.

BUG 2: HandoffLeaseIntegration.transfer() doesn't transfer workers
  Evidence:  Code at app/services/swarm/lease_integration.py:81-96
             (pre-fix) calls self.release(handoff_id) and
             self.claim_for_handoff(handoff_id, to_agent_id), but
             both use self._worker_id.  The from_agent_id and
             to_agent_id parameters are taken but NEVER USED.
  Impact:    Function name + docstring promise cross-worker transfer;
             implementation is a no-op (refreshes same lease).  The
             "passing" integration test only checked lease is not
             None, never verified worker_id actually changed (it
             didn't).
  Fix:       Extracted _claim_for_worker() as the internal claim
             primitive.  transfer() gained new_worker_id: str | None
             = None parameter — if provided, re-claim uses that
             worker_id; if None, uses self._worker_id (backward
             compat preserved).  2 new unit tests verify both paths.

BUG 3: _packet_from_record() crashes on pre-chunk-5 records (NULL budget)
  Evidence:  New unit test test_packet_from_record_old_record_fallback
             constructs a mock HandoffRecord with
             budget_remaining_usd=None, and the call to
             HandoffBudget(remaining=0, initial=0) raises
             pydantic_core.ValidationError: Input should be greater
             than 0 (initial_usd has constraint gt=0).
  Impact:    In production, accept_with_packet on any pre-chunk-5
             handoff record (existing rows with NULL budget) would
             crash with a 500.  The 49dc90f followup added a
             success_criteria fallback but missed the budget case.
  Fix:       In _packet_from_record, when budget_remaining_usd is
             None, set remaining=Decimal('0') and
             initial=Decimal('0.000001') (smallest valid positive
             Decimal satisfying gt=0).  Updated docstring to
             document the fallback.
