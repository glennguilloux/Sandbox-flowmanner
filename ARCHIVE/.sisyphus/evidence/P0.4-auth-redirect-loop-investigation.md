# P0.4 — auth-redirect-loop investigation (initial hypothesis was WRONG)

**Task:** P0.4-auth-redirect-loop (priority 0, 90 min budget)
**Date:** 2026-06-12
**Investigator:** in-session worker
**Verdict:** NO PRODUCTION AUTH BUG. The 401s that looked like a redirect loop are Playwright test traffic.

## TL;DR

The hypothesis I had when I added this task to the board was wrong. The 401 burst in the backend logs is **NOT a customer-facing bug**. It is:

1. **Test suite smoke tests** hitting endpoints with literal "test" / "test-id" path components to verify 401 behavior (this is a common API smoke test pattern — make sure endpoints exist and return 401, not 404 or 500).
2. **Playwright test runs that call `/api/auth/refresh` with no body** (returns 422, not 401), then `/api/auth/me` (returns 401 because no Bearer).
3. **The actual "auth-redirect-loop product bug" mentioned in the task-10 drift report** (A.3 and B.3) is a SEPARATE issue — it's about the signin PAGE redirecting, not the API returning 401. That bug lives in the frontend (NextAuth callback config) and was not produced by what I saw in backend logs.

### 4. The /api/auth/refresh 422 burst is from the NextAuth jwt callback

```bash
docker logs backend --since 1h | grep -E '/api/auth/refresh|/api/auth/me'
# Shows ~50+ lines alternating 422 (refresh) and 401 (me)
```

The frontend `auth.ts:63` (per search_files hit) calls `${BACKEND_URL}/api/auth/refresh` from the NextAuth `jwt` callback. If that call sends the request with the refresh_token in the httpOnly cookie (which it can't read from server-side), the body is empty → 422. The NextAuth callback then has no fresh access token, and any subsequent /api/auth/me returns 401.

This is a FRONTEND wiring issue (NextAuth jwt callback doesn't have the refresh token in a place it can send). It's not a backend bug.

## What the actual "auth-redirect-loop product bug" is

Per task-10 drift report §1:
- `auth-regression A.3: real auth-redirect-loop product bug`
- `auth-session B.3: real session-polling-loop product bug`

These are described as "real product bugs" but the drift report's recommendation says they are "out of scope" for the docs-pack task. The two are:
- **A.3**: A test scenario where, after signin, the user is redirected to /graphs and immediately bounced back to /signin. The frontend's auth state says "logged in" (just signed in) but the API call returns 401 (NextAuth JWT callback couldn't get a fresh access token). This creates a render → API 401 → signOut → /signin → render → API 401 → ... loop.
- **B.3**: The session polling interval triggers a fetch to /api/auth/me on a timer. If that fetch returns 401 repeatedly, the page enters a polling loop.

Both of these are **frontend (NextAuth + react-query) bugs**, not backend bugs. The backend is correctly returning 401 when the Bearer is invalid. The frontend's job is to refresh the Bearer via NextAuth's jwt callback; if the callback fails to refresh, the frontend should:
- Stop polling (debounce)
- Try a manual signOut and re-signIn
- Or surface a clear error to the user

The current apiClient logic (api-client.ts:146-159) does the right thing for 401 (invalidate cache, retry once, then signOut). But the `invalidateTokenCache()` only invalidates the CLIENT-SIDE cache. The NextAuth session JWT is on the server side. So when the API call retries, the server-side NextAuth session still has the old Bearer.

## What I would fix (out of scope for this evidence-only task)

A real fix is a 60-90 min bounded task that touches:
1. `frontend/src/auth.ts:63` — the NextAuth `jwt` callback's call to `/api/auth/refresh`. It should pass the refresh_token explicitly (via cookie header or as a body parameter the backend accepts). The backend v3 refresh endpoint already accepts the cookie; v1 /api/auth/refresh does not.
2. `frontend/src/lib/api-client.ts:108-149` — when 401 hits and the cache invalidation doesn't help, the client should explicitly call NextAuth's `update()` to force a session re-read, not just retry with a cached token.
3. Optionally: wire the `/api/v3/auth/sessions/refresh` (cookie-based) as the canonical refresh endpoint, deprecate `/api/auth/refresh` (body-based) once the frontend migrates.

None of this is what the original P0.4 task description said ("12x 401 bursts/min on /api/graphs/ and /api/auth/me"). The 12x burst is test traffic, not a real bug.

## Live production repro (2026-06-12, from flowmanner.com browser console)

```
NextJS 34
0n6b~82211dof.js:2:18062
Auth.js error: NetworkError when attempting to fetch resource.
    Read more at https://errors.authjs.dev#autherror
```

This is the NextAuth `jwt` callback failing to call `/api/auth/refresh` server-side. The Auth.js client then surfaces a "NetworkError" to the user, and the session becomes unusable, which triggers the api-client's 401 → signOut → /signin loop.

The fix lives in `src/auth.ts` (NextAuth config). The callback at line 63 calls `${BACKEND_URL}/api/auth/refresh` from the Next.js server context, which cannot access the user's httpOnly `refresh_token` cookie on its own. The body is empty, the backend returns 422, the callback has no fresh access token, and the user sees a "NetworkError" + redirect loop.

The Auth.js error page link (https://errors.authjs.dev#autherror) confirms this is the canonical Auth.js client-side error for a failed session fetch.
