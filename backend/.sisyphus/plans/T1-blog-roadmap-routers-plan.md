## 2. Open questions for Glenn (recommended defaults, no fabricated data)

### 2.1 Blog data source — ASK GLENN (no data exists to infer from)

**Facts:** The backend has **no blog model, table, or router**. We do not know whether blog/case-study content is (a) Markdown/MDX in the repo, (b) a CMS (e.g. Strapi/Contentful), or (c) a DB table (new SQLAlchemy model). The `BlogPost` interface shape (`view_count`, `tags` as joined entities, `author_name`, `published_at`) *resembles* a DB row, but that is a UI contract, not proof of a backend source. **We must not invent posts or a schema.**

**Options:**
- (a) Markdown/MDX in repo — zero infra, git-versioned, best for static marketing/content. Requires a loader (e.g. `python-frontmatter` + glob) and loses `view_count`/relational `tags` unless tracked separately.
- (b) CMS — cleanest editorial UX but adds an external dependency + sync; heaviest.
- (c) DB table — matches the `BlogPost`/relational shape best; needs a new model + Alembic migration; self-hosted, no external dependency.

**Recommended default:** **(c) DB-backed new model** if blog content must carry `view_count`/relational tags and be admin-editable without deploys; otherwise **(a) Markdown/MDX in repo** as the lowest-friction default for static content. Lean: since `blog-api.ts` already returns a `BlogListResponse` with `total/limit/offset` (pagination semantics) and `view_count`, the DB model is the most contract-faithful. **Recommend Glenn confirm; do not proceed to implementation until he picks.**

### 2.2 Roadmap: live feed vs legacy dead call — RECOMMEND: live feed

**Facts (not fabricated):** The `roadmap_items` table + `RoadmapItem` model **already exist** in the backend, and the frontend SDK `RoadmapItemOut` matches that model field-for-field. The frontend roadmap page actively calls `GET /api/roadmap/` (page-client.tsx:47). This is **not a dead/legacy call** — it is a live read against an existing, populated table. The model even has `is_public` (default `True`) for filtering.

**Recommended default:** Build `/api/roadmap` (+ `/api/roadmap/`) as a **live, read-only GET feed** off `roadmap_items WHERE is_public = true`, returning `RoadmapItemOut[]`. No new model needed for the items list.

**Gap to resolve (sub-question, flag for Glenn):** `/api/roadmap/categories` expects `RoadmapCategoryOut {id, name, count}`, but there is **no `RoadmapCategory` model** in the backend. Two clean fixes:
- (i) **Derive** categories with `SELECT category, COUNT(*) FROM roadmap_items WHERE is_public GROUP BY category` (no new table) — recommended, zero-schema.
- (ii) Add a `roadmap_categories` table + model (only if Glenn wants editable category metadata).

**Recommend Glenn take (i) by default** unless he wants managed categories.

### 2.3 Emergent: blog auth boundary conflict — ASK / RECOMMEND

**Conflict found:** `blog-api.ts` raw-fetches `/api/blog/posts` and `/api/blog/posts/{slug}` **anonymously** (no token, ISR `revalidate:60`), implying these are *public content*. But the `apiClient` security contract (`api-client.test.ts:84–89`) asserts `/api/blog/posts` and `/api/blog/feed` are **auth-required** (children of the exact-only public parent `/api/blog`). The roadmap page goes through the SDK which *does* send a token, so its auth-gated children are consistent; the blog raw-fetch is the outlier.

**Options:**
- (A) Make blog read endpoints **public** (they are marketing content). Then move them to `PUBLIC_PREFIX_PATHS` / `PUBLIC_SUBPATHS` in `api-client.ts` and relax the two `api-client.test.ts` guards (lines 84–89). Keep `blog-api.ts` as-is.
- (B) Keep blog read endpoints **auth-required** (honor the existing guard). Then `blog-api.ts` must be migrated to `apiClient` so it injects the token (new task; do not bundle into T1).

**Recommended default:** **(A) public blog reads** — a public blog/case-study surface should not require login, and `blog-api.ts`'s anonymous ISR fetch confirms that was the original intent. Flag the `api-client.test.ts` guard update as a required companion change.

### 2.4 Emergent: v1-path vs v2-envelope mismatch — FLAG (affects frontend, not just backend)

**Finding (important):** The current frontend calls **`/api/blog` and `/api/roadmap`** (unversioned, v1-style paths) and expects **raw, un-enveloped shapes**:
- Roadmap SDK: `RoadmapItemOut[]` (a bare array) and even sends `Accept-Version: v1`.
- Blog: `BlogListResponse {posts,total,limit,offset}` (not the v2 `paginated` envelope).

This task (and source plan §T1 step 3) specifies **v2 enveloped** routers under `/api/v2/...` returning `{data, meta, error:null}`. If we build `/api/v2/blog` + `/api/v2/roadmap` enveloped, the **existing frontend consumers will not parse them** without a frontend migration (new paths + envelope unwrap). That work is out of T1's scope (T2/T3 in the source plan are separate).

**Options to confirm with Glenn:**
- (i) Build **v2 enveloped** routers as specified (this plan), and add a **frontend follow-up task** to repoint `blog-api.ts` + `RoadmapService` to `/api/v2/...` and unwrap the envelope. Cleanest long-term; matches "v2 is the default."
- (ii) Build **v1-style (un-enveloped)** routers at `/api/blog` + `/api/roadmap` to match current consumers exactly (violates the "v2 default" policy; only if Glenn wants zero frontend change now).
- (iii) Build v2 enveloped AND add a thin **version shim** so `Accept-Version: v1` still returns the raw shape. More work, maximal compatibility.

**Recommended default:** **(i)** — follow the v2 envelope contract as the task specifies, and spawn a small frontend migration task to consume it. Do not silently ship enveloped responses to the current raw-shape consumers.
