## 2. Why Squashing Is Low-Risk *in Theory* (but still not free)

A squash = replace the 149 revisions with a handful of "baseline" revisions
that `CREATE TABLE`/ `ALTER` the schema as it stands today, then:
1. Stamp the `alembic_version` table on every environment to the new head.
2. Delete the old 149 files.

Why it is safer than the usual horror story:
- **One head, 4 known roots, all merges resolved** → no ambiguous
  branch-label divergence to preserve.
- **No custom version table** → stamping is a one-line `alembic stamp` per env.
- The schema is already converged, so a freshly-generated squash would
  reproduce the current state exactly.

---

## 3. The Real Risks (the 15 data-mutation files are the danger)

This is NOT a pure-DDL squash. **15 migrations perform data mutation**
(backfills, sentinel UPDATEs per the 2026-06-25 data-mutation convention,
re-encryption of BYOK keys, etc.). Squashing DROPS these downgrade/upgrade
data paths:

1. **Rollback integrity loss.** If a future `alembic downgrade` needs to step
   *through* a data-mutation revision, that logic no longer exists after a
   squash. The 2026-06-25 convention ("use UPDATE with sentinel, never DELETE")
   exists precisely because some of these were destructive-adjacent.
2. **BYOK re-encryption (`20260630_plan_candidates` lineage / per-key salt).**
   Re-encrypting API keys is **non-idempotent and security-sensitive** — a
   squash that bakes in "already encrypted" state breaks any environment that
   has NOT yet run that migration.
3. **Cross-environment drift.** Multiple deployed DBs (homelab + VPS + any
   staging) are pinned at *different* points in the 149-chain. A global squash
   requires stamping **each** to the new head with zero missed env, or that env
   will attempt to re-run/ skip incorrectly on next deploy.
4. **`alembic_version` version-number collision.** If any future migration is
   autogenerated against the OLD chain by a developer who didn't pull the
   squash, you get a second head → the exact branching problem this card
   wanted to avoid. Requires a coordinated flag-day.
5. **The 226 KB `reconcile_schema_001_additions.py`** is a reconciliation
   migration (probably hand-fixing drift). Squashing it hides the audit trail
   of *why* columns were added — losing forensics.

## 4. Recommended Approach (if you ever do it)

**Do NOT do it in this card.** This is a PROPOSAL-only task. If approved later:

**A. Preferred: "baseline + prune", not "squash-in-place"**
   1. Generate a single new migration `0001_baseline_schema.py` that emits the
      *current* DDL (via `alembic revision --autogenerate` against an empty DB,
      then verify the resulting schema matches prod with `pg_dump --schema-only`
      diff).
   2. Keep it as a NEW head alongside the old chain; do **not** delete the 149
      yet. Run both in CI for one release.
   3. Only after a clean deploy window, switch `alembic_version` and retire the
      old files in a *separate* reviewed PR.

**B. Stamp procedure per environment (must be exhaustive):**
   ```
   docker compose exec backend alembic stamp <new_baseline_head>
   ```
   Run on homelab, VPS, and every staging DB. Verify with
   `alembic current` before the next `deploy-backend.sh`.

**C. Preserve the 15 data-mutation migrations' *intent* in a `docs/` note**
   (which envs have run BYOK re-encryption, which have backfills applied) so a
   fresh DB from baseline is known-complete.

**D. Guardrail:** add a CI check (`alembic check` / `alembic heads` == 1) so a
   stray second head fails the build.

## 5. Recommendation to Glenn

- **Severity was correctly rated LOW.** The 149 files cause: (a) slower
  `alembic history`/`upgrade` graph-walk startup (sub-second, negligible), (b)
  noisy diffs, (c) occasional merge conflicts on `alembic_version` when two
  agents autogenerate concurrently.
- **None of these justify the rollback/security risk of a squash today.**
- **Cheaper mitigations that capture most of the benefit:**
  1. Keep the existing ruff exclusion (already done).
  2. Add a CI `alembic heads` == 1 guard to prevent *future* branch drift.
  3. Periodically run `alembic history` in PR reviews; only squash when the
     chain exceeds ~250 files AND all envs are known-converged.
- **Propose closing this card as "resolved-by-proposal / won't-fix-now"** with
  the guardrail above adopted. Re-open only if file count or deploy friction
  actually grows.
